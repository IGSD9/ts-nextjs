## Step 03 — Prisma とデータベーススキーマ

| 項目 | 内容 |
|------|------|
| 目的 | 要件のモデルを物理スキーマとして確定する |
| 入力 | 基本設計の ER |
| 処理 | ① `schema.prisma` に `User`, `DailyReport`, `PositiveMessage` を定義 ② `DailyReport` に `wednesdayExtra Int?`（または `extraNote` 等、水曜ライト質問用）を追加 ③ `DailyReport` に `@@unique([userId, reportDate])` を検討（`reportDate` は日付のみの `DateTime @db.Date` など）— **1日1件** を DB で保証 ④ `npx prisma migrate dev` |
| 出力 | マイグレーション済み DB |
| 確認 | Prisma Studio でテーブルが見える |

**補足（1日複数回チェックイン）**: 要件が「上書き」なら `upsert` で同一 `userId`+`reportDate` を更新。禁止ならユニーク制約＋エラー表示。

## Step 05 — Supabase Auth（マジックリンク）フロー

| 項目 | 内容 |
|------|------|
| 目的 | F-01, F-02 を満たす |
| 入力 | メールアドレス、Supabase Auth 設定 |
| 処理 | ① `SC-01` でメール入力 → `signInWithOtp`（またはプロジェクト方針の API）② メールのリダイレクト URL を `SC-02` に設定 ③ `SC-02` でセッション確立後、**サーバー**で `auth.users` のメールと `User` を照合 ④ 未登録なら `User` を `create`（name 初期値はメールのローカル部など） |
| 出力 | ログイン済みセッション＋`User` レコード |
| 確認 | 新規メールで初回のみ User が増える |

## Step 07 — チェックイン画面（SC-03）

| 項目 | 内容 |
|------|------|
| 目的 | F-04 を満たす（3秒 UX） |
| 入力 | 6項目＋水曜のみ追加 UI |
| 処理 | ① 各項目を大きなタップ領域で表示（補助文は要件どおり）② 水曜判定は `getDay() === 3`（ローカルタイムゾーン方針をコメントで固定）③ 送信は Server Action ④ `DailyReport` を当日分 `upsert` ⑤ 成功時 `SC-04` へ `redirect` |
| 出力 | 保存済みレポート |
| 確認 | 同一日に再送信で上書きまたはエラー（Step 03 の方針どおり） |

## Step 08 — チェックイン完了（SC-04）— メッセージ＆ピザ

| 項目 | 内容 |
|------|------|
| 目的 | F-06, F-07 の一部 |
| 入力 | なし（または直近送信フラグ） |
| 処理 | ① `PositiveMessage` を `count` し 0 件ならメッセージ枠はスキップ or 固定文 ② ランダム1件: `skip = floor(random() * count)` 等（SQL またはアプリ乱数）③ ピザメーター: 当日チェックイン人数／登録ユーザー数、および直近7日の平均「元気スコア」（簡易式: `6 - fog` の週平均など）から **0–100 のゲージ** を算出し表示 ④ 100 到達時にバッジ演出（ローカルストレージで「今週表示済み」等を避けるかは製品判断） |
| 出力 | 完了 UI |
| 確認 | メッセージがランダムに変わる（件数十分時） |

## Step 11 — FOG アラート集計（管理者）

| 項目 | 内容 |
|------|------|
| 目的 | BR-01, F-08 の一部 |
| 入力 | 全ユーザーの `DailyReport`（直近十分な期間） |
| 処理 | **推奨定義（本書で固定）**: 「各ユーザについて、チェックインした日を日付順に並べ、直近から遡り `fog === 3` が **5回連続**（連続するチェックイン日）ならアラート」— 欠席日は連続カウントに含めない。別案「カレンダー5日連続」にする場合は基本設計と齟齬が出るため変更時は両文書を更新 |
| 出力 | アラート対象ユーザー一覧（メール／名前は管理者のみ表示可） |
| 確認 | テストデータで 5 連続濃霧ユーザーのみ表示 |
